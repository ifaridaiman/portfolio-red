# ADR-0005: RAG with pgvector in PostgreSQL

- **Status:** Accepted
- **Date:** 2026-07-01

## Context

The digital twin must answer questions grounded in published portfolio content with source citations. We need vector similarity search, operational simplicity, and alignment with the Prisma/PostgreSQL data layer (ADR-0004). Managing a separate vector database adds infrastructure cost for a single-tenant portfolio.

## Decision

Implement **RAG retrieval using pgvector** in the same PostgreSQL instance:

- Store embeddings in `KnowledgeChunk` (or equivalent) table with `vector` column
- Use cosine distance (`<=>`) for similarity search
- Embed via provider API (default OpenAI `text-embedding-3-small`)
- Ingest on content publish from projects and articles
- AI Gateway assembles retrieved chunks into prompts (see [rag.md](../01-architecture/rag.md))

## Consequences

### Positive

- One database to backup, migrate, and monitor
- Transactional consistency between content and chunks
- Lower operational overhead than separate vector SaaS for v1
- Fits self-hosted Docker/EC2 deployment

### Negative

- Vector index tuning (IVFFlat/HNSW) requires DBA attention at scale
- Very large corpora may outgrow single-DB performance
- Prisma may need raw SQL for vector queries initially

### Neutral

- Hybrid full-text + vector search deferred to future improvement

## Alternatives

### Alternative 1: Pinecone / Weaviate / Qdrant (managed vector DB)

- **Pros:** Optimized ANN search at scale
- **Cons:** Extra service, cost, sync pipeline with Postgres content
- **Why rejected:** Overkill for single-owner portfolio v1

### Alternative 2: Elasticsearch/OpenSearch hybrid

- **Pros:** Strong keyword + vector hybrid
- **Cons:** Heavy ops burden for small team
- **Why rejected:** Operational complexity

### Alternative 3: In-memory vector search (e.g., hnswlib on boot)

- **Pros:** Fast prototype
- **Cons:** Not durable; poor fit for serverless/multi-instance deploy
- **Why rejected:** Production needs persistent index
